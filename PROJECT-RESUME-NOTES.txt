PROJECT NAME
- Boitekong Pulse

PROJECT GOAL
- Boitekong Pulse is a mobile-first township community web app.
- It is built for local updates, replies, comments, direct messages, voice notes, notifications, search, and community reporting.
- The product direction is: local, township-first, voice-friendly, and community-centered.

CURRENT APP STATE
- The app is currently a frontend-first prototype.
- Data is still stored in browser localStorage.
- There is no real backend yet.
- There is no real multi-device sync yet.
- There is no real push notification backend yet.
- Reports exist as submission flow only, not as moderator UI.
- Voice notes exist in the prototype and are currently stored in frontend/browser memory/local storage patterns.

CURRENT PHASE
- We are at the end of Phase 6.
- Phase 6 = frontend feature build + UX polish + cleanup/stabilization.
- Phase 7 should begin next.

RECOMMENDED VERSION LABELS
- Current internal status: Phase 6 complete-ish / frontend prototype mature.
- Next step: Phase 7 = backend foundation.
- Suggested next milestone labels:
  - Phase 7 / v0.7 = backend scaffold + auth/users
  - Phase 8 / v0.8 = posts/comments/messages/notifications API migration
  - Phase 9 / v0.9 = moderation/report backend + OTP + real media handling
  - Beta / v1.0-beta = responsive sweep + manual QA + deployment prep

WHAT HAS ALREADY BEEN BUILT

1. AUTH
- Login page exists.
- Register page exists.
- Login accepts either username or phone number in one input.
- Register requires phone number.
- Register has a 16+ checkbox.
- Auth screens were redesigned closer to Instagram style:
  - floating placeholder labels
  - required-field red state
  - cleaner CTA/buttons
  - footer area present
- Brand copy and naming now use "Boitekong Pulse."

2. PROFILE
- Profile view exists.
- Edit profile exists.
- User can update:
  - username
  - phone number
  - avatar
  - township
  - extension
  - password
- Invite button exists in profile:
  - uses Web Share API when available
  - clipboard fallback otherwise

3. FEED / POSTS
- Feed exists.
- Text posts exist.
- Voice-note posts exist.
- Create post exists with two modes:
  - Text update
  - Voice note
- Voice-note post length is capped at 1 minute.
- Create post no longer asks for image/township/extension.
- B-Point location is taken from the current user profile.
- Edit post now matches the new post model better:
  - text-only editing
  - voice-note posts cannot be edited
  - voice-note posts must be deleted and reposted
- Post owners can delete their own posts from the 3-dot menu.

4. COMMENTS / REPLIES
- Comments exist.
- Replies exist.
- Text comments are capped at 300 characters.
- Voice-note comments and replies exist.
- Report action exists for comments and replies.
- Delete/edit own comment/reply flow exists.
- Comment/reply voice-note styling has been brought closer to DM styling.
- Comments and replies support batching / "See more."

5. DIRECT MESSAGES / MESSENGER
- Messages screen exists.
- User discovery in Messages exists.
- Search people by name/location exists in Messages.
- DM toggle exists:
  - enable/disable direct messages
- Block / unblock exists from conversation actions.
- Sending rules exist:
  - if I disabled DMs, I cannot send
  - if the other user disabled DMs, I cannot send
  - if blocked either way, messaging is blocked
- Text messages exist.
- Voice-note messages exist.
- DM voice notes are capped at 1 minute.
- Voice-note UX in DMs has been heavily polished:
  - recorder state
  - preview state
  - posted bubble state
  - playback speed retained
- Message edit/delete behavior exists.
- Privacy-friendly DM previews were added:
  - inbox preview does not reveal deleted content
  - notifications do not reveal message body text

6. NOTIFICATIONS
- In-app notifications exist.
- Notification toggle exists:
  - user can turn notifications on/off
- Desktop/browser notifications were added for supported browsers.
- Notification click behavior is wired:
  - DM -> opens conversation
  - follow -> opens user preview
  - comment -> opens relevant post/comments
  - reply -> opens relevant post/comments
- Follow notifications exist.
- Comment-on-post notifications exist.
- Reply-to-comment notifications exist.
- Notifications respect hidden reported content for the reporting user.

7. SEARCH
- Search screen exists.
- User search exists.
- Post search exists.
- Search batching / "See more" exists.

8. REPORTING
- Post report flow exists.
- Comment/reply report flow exists.
- Preset reasons exist.
- "Other" note is limited to 120 characters.
- Duplicate reports from the same user on the same target are blocked.
- Optional "hide for me" exists:
  - hidden reported post/comment is removed from that reporter's view
  - related notifications are also suppressed for that same user
- Moderator UI is NOT part of the frontend plan right now.

9. UI / POLISH / BRAND
- Brand renamed from "Boitekong Now" to "Boitekong Pulse."
- App icon / logo now use the shared asset in assets/app-icon.png.
- Login showcase and auth pages have been polished heavily.
- Side drawer cleanup is done:
  - removed username/avatar/B-Point block
  - install button added
  - contact overflow fixed
- Global scrollbar, side drawer scrollbar, and comments panel scrollbar were hidden.
- Top navbar now touches the top and side edges even on larger viewports.

BACKEND PHASE - WHAT WE SHOULD BUILD NEXT

PHASE 7 GOAL
- Move from localStorage prototype to real backend foundation.
- Keep current UX and current frontend data flows where possible.
- Replace localStorage service internals with API-backed versions over time.

RECOMMENDED BACKEND STACK
- Node.js
- Express
- MongoDB
- Mongoose
- JWT or session-based auth
- SMS OTP provider later for phone verification
- Cloud storage / media service for avatars, images, and voice notes

WHY MEDIA STORAGE MATTERS
- We should NOT keep voice notes and image payloads as base64 in the long-term backend.
- Backend should store file metadata + URLs, not giant data URLs in documents.
- Use object storage or media hosting for:
  - profile avatars
  - post images if reintroduced or supported later
  - voice notes in posts
  - voice notes in comments/replies
  - voice notes in direct messages
