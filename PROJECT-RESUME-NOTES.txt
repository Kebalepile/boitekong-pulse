PROJECT NAME
- Boitekong Pulse

PROJECT GOAL
- Boitekong Pulse is a mobile-first township community web app.
- It focuses on local updates, comments, replies, direct messages, voice notes, notifications, search, and reporting.
- Product direction remains: local, township-first, voice-friendly, and community-centered.

CURRENT APP STATE
- The app is no longer just a frontend-only prototype.
- It is now a hybrid web app with a real backend plus frontend caching.
- The frontend service layer is already API-backed for:
  - auth
  - user profile/settings/search/follow/block
  - posts/comments/reactions
  - conversations/messages
  - notifications
  - reports
- Browser localStorage is still used for:
  - access token/session persistence
  - cached users/posts/conversations/notifications/reports
  - hidden-report state
  - direct-message encryption keys
  - lightweight UI preferences
- Backend now includes:
  - Express app
  - MongoDB/Mongoose
  - JWT auth
  - SMS OTP flow
  - posts/comments API
  - conversation/message API
  - notifications API
  - reports API
  - realtime websocket sync at `/api/realtime`
- Browser notifications exist, but full push-notification infrastructure is NOT done.
- Voice notes are stored through the current Mongo/base64-binary migration flow for now.
- Dedicated media/object storage is still a later step.

WHAT IS ALREADY BUILT

1. AUTH + ACCOUNT
- Register and login use the API.
- Login accepts username or phone number.
- Register requires phone number.
- Forgot password now exists from the login page:
  - user enters the registered phone number
  - SMS reset OTP is sent only if that number exists in the database
  - reset OTP expires after 5 minutes
  - forgot-password reset can only be completed once every 24 hours
  - successful reset returns the user to the login page
- Profile update uses the API.
- Phone verification flow exists in code:
  - send OTP
  - verify OTP
- Direct-message encryption key sync exists.

2. PROFILE + COMMUNITY GRAPH
- Profile page is built.
- Edit profile is built.
- Follow/unfollow is API-backed.
- Followers/following lists are API-backed.
- DM and notification settings are API-backed.
- Block/unblock is API-backed.

3. FEED / POSTS / COMMENTS
- Feed loads from the API.
- Create post is API-backed.
- Edit/delete post is API-backed.
- Post reactions are API-backed.
- Comments/replies are API-backed.
- Comment reactions are API-backed.
- Voice-note posts/comments/replies are supported in the current migration flow.

4. DIRECT MESSAGES
- Conversations load from the API.
- Direct conversation creation is API-backed.
- Send/edit/delete message is API-backed.
- Voice-note messages are supported.
- Conversation archive/delete-from-inbox behavior is API-backed.
- Read state is API-backed.
- End-to-end encryption session handling exists on the frontend.

5. NOTIFICATIONS
- Notification list is API-backed.
- Mark one/all as read is API-backed.
- Conversation notifications can be cleared/read.
- Browser notifications are wired for supported browsers.
- Notification click routing is wired in the app.

6. SEARCH
- User search is API-backed.
- Post search is API-backed.
- Search screens still use cache as a safe fallback if refresh fails.

7. REPORTING
- Report submission is API-backed.
- Duplicate report protection exists.
- Hide-for-me flow exists.
- Hidden reported content is filtered in feed/comments/notifications for the reporting user.
- Moderator UI is still postponed.

8. LIVE SYNC
- Polling plus websocket live sync exists.
- Feed can opt into live post refresh.
- Messages and notifications can refresh from realtime events.
